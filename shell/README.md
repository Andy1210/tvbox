# tvbox - Electron shell

The host process for the TV box. A single fullscreen, always-on-top **Electron**
window that:

- boots into the **HOME launcher** (the React app in `../launcher`, served under
  `/tvbox/`),
- hosts **apps** installed as self-contained packages under
  `~/.tvbox/apps/<id>/` (a `manifest.json`, a `web/` UI bundle, and an optional
  host `plugin.js`), fetched from the
  [tvbox-apps registry](https://github.com/Andy1210/tvbox-apps) - the shell
  itself ships no first-party apps, only the SDK and this host. Local bundles
  (e.g. Plex HTPC's official 10-foot UI) are served from their install and
  composited over **mpv**, which plays video BEHIND the transparent window,
- gives each app a **capability-scoped bridge** (`preload.js` in the main
  window, `preload-app.js` in the hardened isolated window),
- and routes the remote: the **Home** button returns to the launcher from any
  app; Back / media / transport keys reach the focused app.

Native ARM, no emulation.

## Layout

```
shell/
  main.js               Electron host: HTTP server + route dispatch, manifest registry +
                        install runner, mpv control, window, nav, capability IPC, plugin loader
  install.js            manifest loading + validation + install-recipe runner (shared with cli.js)
  cli.js                the `tvbox` CLI (list/deps/install/remove/update/backup/restore)
  preload.js            main-window bridge (launcher + local static bundles; Node-capable)
  preload-app.js        isolated-window bridge for capability apps (contextBridge, sandboxed)
  appfetch.js/appdata.js  capability brokers: origin-locked fetch proxy + per-app storage
  service_supervisor.js generic supervised child process (capped backoff + give-up)
  audio-default.sh      runtime HDMI sink detection (TV/port-independent)
  bridges/<name>.js     renderer bridge adapters, manifest-selected (runtime.bridge; e.g. qwebchannel for Plex)
  launcher-dist/        built React launcher (generated by `../launcher` build; gitignored)
```

There is no `shell/apps/` or `shell/plugins/` directory: apps are **packages**
installed under `~/.tvbox/apps/<id>/` (manifest + `web/` UI + optional
`plugin.js`) from the registry, and a package ships its own plugin.

Installed web-client bundles land in **`~/.tvbox/apps-data/<id>/`** (outside the
shell install, next to `~/.tvbox/apps/`), so they survive OTA updates and
deploys - an OTA runs a fresh `~/.tvbox/current/shell`, which never carries
them. `install.js` migrates any bundle from the old in-shell location on first
load.

## The app model (manifest)

Each app is a package under `~/.tvbox/apps/<id>/`: a `~/.tvbox/apps/<id>/manifest.json`
alongside its `web/` UI and an optional `plugin.js`. A manifest-only app can
also be a bare `~/.tvbox/apps/<id>.json`. Packages survive deploys and OTA
updates. Full field reference + JSON Schema:
[docs/app-manifest.md](../docs/app-manifest.md). The launcher reads the tile
fields via `GET /tvbox/api/apps` (manifests are re-read on each call, so a
freshly dropped-in manifest appears live); the shell consumes `install` +
`runtime`.

```jsonc
{
  "id": "plex",
  "name": "Plex", // or { "hu": "Élő TV", "en": "Live TV" }
  "type": "webclient", // the only type - apps are web packages the shell serves/loads
  "status": "ready", // "ready" | "coming_soon"
  "accent": "#e5a00d",
  "icon": "plex",
  "tagline": { "hu": "Filmek és sorozatok", "en": "Movies & TV shows" },
  "install": {
    // web-client install recipe (Homebrew-like)
    "source": { "type": "flatpak", "ref": "tv.plex.PlexHTPC", "arch": "x86_64" },
    "extract": "files/resources/web-client",
    "patch": [{ "op": "strip-script", "match": "qwebchannel" }],
  },
  "runtime": {
    "serve": "static",
    "entry": "index.html",
    "mount": "root", // served at "/"
    "capabilities": ["storage", "player", "input", "system"],
    "player": "mpv-overlay",
  },
}
```

`capabilities` is the security/extensibility boundary: `preload.js` only injects
the matching bridge surface (the launcher gets `nav` only; a web-client app also
gets the QWebChannel `storage` + mpv-backed `player` + remote `input`). Source
types beyond `flatpak` (url tarball, git) arrive with the `tvbox install` CLI.

`requires.bin` lists the runtime binaries the app needs (e.g. `mpv`, `librespot`).
The shell resolves them on PATH (`install.appDeps`) and the launcher greys the tile

- shows "needs X" when one is missing - nothing crash-loops for an absent binary.

## Shell-side plugins (`service`)

YouTube is a pure manifest (a remote site). An app that needs shell-side logic - a
long-lived daemon, an OAuth window, custom HTTP routes, cast/standby hooks -
declares `"service": "<name>"` and ships a **`plugin.js` inside its own package**
(`~/.tvbox/apps/<id>/plugin.js`). **Spotify** (librespot daemon + OAuth + cast)
and **Live TV** (the IPTV data proxy routes) are plugins. Plugins run in the
**host** process, not the renderer.

The shell loads a plugin **only when its app is present AND its `requires.bin`
resolve** (so a box without `librespot` never touches the Spotify subsystem - no
routes, no daemon, no respawn loop; a box without `mpv` never loads the Live TV
routes). The `service` name is validated `^[a-z0-9_-]+$`, and the plugin is
required from the app package's own directory (`m._dir/plugin.js`) - the shell
has no first-party `plugins/` directory.

A plugin is a factory `(host) => ({ start?, stop? })` that registers its routes via
`host.registerRoutes` when the factory runs (synchronously, before the server
starts):

```js
// ~/.tvbox/apps/spotify/plugin.js
module.exports = (host) => {
  // ...closures over host...
  host.registerRoutes("/tvbox/api/spotify", {
    "GET /state": (req, res) => host.json(res, state()),
    "GET /stream": (req, res) => {
      /* SSE */
    },
    "POST /event": (req, res, ctx) => {
      handle(ctx.body);
      host.json(res, { ok: true });
    },
  });
  return {
    start() {
      host.spawnService("librespot", spec); /* wire cast + pairing */
    },
    stop() {
      host.stopService("librespot");
    },
  };
};
```

Route handlers get `(req, res, ctx)`; `ctx.body` is the parsed JSON for POSTs. The
generic server tries registered plugin routes before its own built-ins. `host` is a
deliberately small, scoped surface - a plugin never sees `win`, `mpv`, or the
manifest registry:

| `host.…`                                                      | what                                                                |
| ------------------------------------------------------------- | ------------------------------------------------------------------- |
| `registerRoutes(prefix, table)`                               | mount routes keyed `"METHOD /subpath"` under a prefix               |
| `config`                                                      | config store (`rawSpotify` / `setSpotify` / `publicConfig`)         |
| `pairing`                                                     | phone-keyboard `onKey` / `onFill` wiring                            |
| `BrowserWindow`                                               | Electron class, for a plugin that opens its own window (OAuth)      |
| `spawnService(name, spec)` / `stopService` / `restartService` | supervised child (backoff + give-up)                                |
| `showLauncher(hash)`                                          | stop other playback + bring the launcher forward (optional `#view`) |
| `audioSink()`                                                 | detected HDMI sink `node.name`                                      |
| `childEnv()`                                                  | spawn env with the session's Wayland vars                           |
| `json(res, obj)` / `log(…)` / `base`                          | small helpers                                                       |

`spawnService`'s `spec` is `{ argv:()=>[bin,...], env, stdio, minUptimeMs, ceiling,
onGiveUp, log }`: `argv` is recomputed on each (re)start so runtime values (a
renamed Connect device, the detected sink) stay fresh, and a missing/broken binary
logs one line and gives up instead of ENOENT-looping.

Routes stay at their historical `/tvbox/api/<id>/*` paths where an external contract
demands it - Spotify keeps `/tvbox/api/spotify/*` because the OAuth redirect URI is
registered verbatim in the user's Spotify dashboard.

## Apps are opt-in; deps are dep-gated

`deploy.sh` installs only a **baseline** (Electron, Wayland/CEC/audio, node,
flatpak). No app binaries. On a fresh box **only YouTube works** (a remote site,
zero deps); every other app is opt-in and its tile is greyed until its deps resolve.

- **Binary deps** (`mpv` for Live TV/Plex, `librespot` for Spotify) install with
  **`tvbox deps <id>`** on the box. A manifest that offers a prebuilt static
  build (`requires.download`: per-arch url + sha256 → `~/.tvbox/bin`, which is
  on PATH) installs with **no root at all**; `requires.apt` deps (+ any
  third-party repo from `requires.aptRepo`, then disabling a
  `requires.disableService`) are the one step that asks for sudo. Root never
  reaches the running shell (no root-capable HTTP endpoint).
- **Bundles** (Plex's flatpak, url/git web clients) install **user-space, from the
  UI**: selecting a not-yet-installed app POSTs `/tvbox/api/apps/install`, which runs
  `cli.js install <id>` out of process (`flatpak --user`, so no root; never blocks
  the main process) - the launcher polls `/tvbox/api/apps` until `installed` flips.
  Same recipe from the CLI: `tvbox install <id>`.
- **mpv** is a shared, lazy, **dep-gated** player service: spawned only when a
  player-capable app actually plays and only if `mpv` is on `PATH`; otherwise it
  emits an error event instead of an ENOENT spawn. It is never a mandatory boot
  service.

Key learnings baked in:

- mpv "default" audio device = "no target node" here, so we **auto-detect the HDMI
  sink at runtime** (`audio-default.sh`) and pass it explicitly.
- `player.play` is **idempotent** (resume if the same url is already loaded).
- `player.seekTo` arg is **milliseconds** (÷1000 for mpv).
- transparency is a SYNCHRONOUS `tvbox-video` class toggle (no insertCSS race).
- `userData` is pinned to `~/.tvbox/shell-userdata` so app logins survive renames.
- remote: Back = `Backspace`, Stop = `Cancel`, Home = `BrowserHome` (→ launcher).

## Run (deployed to ~/.tvbox/shell)

```sh
npm install   # once (downloads Electron ARM64)
ELECTRON_OZONE_PLATFORM_HINT=wayland XDG_RUNTIME_DIR=/run/user/1000 WAYLAND_DISPLAY=wayland-0 \
  ./node_modules/.bin/electron . --ozone-platform=wayland --no-sandbox
```

Deployed + boot-wired by [`../deploy/deploy.sh`](../deploy/deploy.sh); needs the
CEC→uinput remote bridge (`../cec/cec_uinput_bridge.py`, installed as the
`tvbox-cec` systemd **user** service).
